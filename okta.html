1. What Happens After an Okta Password Change

When a user changes their Okta password, the following can break:

SymptomRoot Cause
Office apps (Word, Excel, Outlook) keep prompting for sign-inCached OAuth token is expired/invalid
OneDrive stops syncing, freezes, or crashes ExplorerOneDrive holds a stale Okta token and loops trying to refresh it
High CPU/memory usage after password changeOneDrive sync process is stuck in a retry loop
Teams shows "You're signed out" or goes offlineTeams token tied to Okta SSO is invalidated
Browser apps still work but desktop apps don'tBrowser picks up new Okta session cookie; desktop apps use cached tokens

Root cause: Office desktop apps and OneDrive cache OAuth tokens independently of the browser. After an Okta password change, these cached tokens become invalid but the apps keep retrying silently — causing CPU spikes, crashes, and sync failures.

2. Step 1 — Re-Authenticate via Okta Dashboard

Do this first. The Okta dashboard is the authoritative source for passing a fresh token to connected apps.

StepAction
1Open a browser and go to your Okta org URL (e.g., https://yourcompany.okta.com)
2Sign in with the new password
3Once logged in, click your name/avatar > Settings
4Scroll to Security > Sessions and click Sign out all devices to clear all stale sessions
5Sign back in to the Okta dashboard with new credentials
6Launch Office 365 from the Okta dashboard (via the app tile) — this passes a fresh SAML/OIDC token to the app
7Repeat for OneDrive, Teams, and any other tiles

Why this works: Launching apps from the Okta dashboard initiates a new SSO assertion, passing a fresh token directly to the app rather than relying on the stale cached token.

3. Step 2 — Fix Office Apps Token (Word, Excel, Outlook, Teams)

Windows — Sign Out and Back In

StepAction
1Open any Office app (Word, Excel, Outlook)
2Go to File > Account
3Click Sign Out under the user's name
4Close all Office apps completely (check Task Manager — kill any remaining Office processes)
5Reopen the Office app
6Sign in with the new Okta-linked Microsoft credentials
7If prompted with MFA, complete it

Kill All Office Processes (PowerShell)

Get-Process -Name WINWORD, EXCEL, OUTLOOK, TEAMS, ONENOTE, MSACCESS -ErrorAction SilentlyContinue | Stop-Process -Force

Clear Office Token Cache (PowerShell — Run as Admin)

# Clears cached Office identity tokens
Remove-Item -Path "$env:LOCALAPPDATA\Microsoft\Office\16.0\Licensing" -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item -Path "$env:LOCALAPPDATA\Microsoft\identitycache" -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item -Path "$env:APPDATA\Microsoft\Protect" -Recurse -Force -ErrorAction SilentlyContinue
Write-Host "Office token cache cleared. Reopen Office apps and sign in."

After running: Reopen Office apps and sign in fresh via Okta dashboard tile or with Microsoft credentials linked to Okta.

Per-App Reset Commands

Use these when a specific app is still hanging after the generic cache clear above.

Outlook — Stop & Clear Auth Cache

Stop-Process -Name OUTLOOK -Force -ErrorAction SilentlyContinue
Remove-Item "$env:LOCALAPPDATA\Microsoft\Office\16.0\Licensing" -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "$env:LOCALAPPDATA\Microsoft\identitycache" -Recurse -Force -ErrorAction SilentlyContinue
Write-Host "Outlook cache cleared. Reopen and sign in."

Still looping? Try Start-Process outlook.exe -ArgumentList "/resetnavpane" to reset the nav pane, or go to File > Account > Sign Out and sign back in.

Teams (Classic) — Stop & Clear Cache

Stop-Process -Name Teams -Force -ErrorAction SilentlyContinue
$teamsCache = "$env:APPDATA\Microsoft\Teams"
Remove-Item "$teamsCache\Cache"         -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "$teamsCache\blob_storage"  -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "$teamsCache\databases"     -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "$teamsCache\GPUCache"      -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "$teamsCache\IndexedDB"     -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "$teamsCache\Local Storage" -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "$teamsCache\tmp"           -Recurse -Force -ErrorAction SilentlyContinue
Write-Host "Teams (Classic) cache cleared. Restart Teams and sign in."

Teams (New / Teams 2.0) — Stop & Clear Cache

Stop-Process -Name ms-teams -Force -ErrorAction SilentlyContinue
Remove-Item "$env:LOCALAPPDATA\Packages\MSTeams_8wekyb3d8bbwe\LocalCache\Microsoft\MSTeams" -Recurse -Force -ErrorAction SilentlyContinue
Write-Host "New Teams cache cleared. Restart Teams and sign in."

Word / Excel — Stop & Clear Token Cache

Stop-Process -Name WINWORD,EXCEL -Force -ErrorAction SilentlyContinue
Remove-Item "$env:LOCALAPPDATA\Microsoft\Office\16.0\Licensing" -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item "$env:LOCALAPPDATA\Microsoft\identitycache"         -Recurse -Force -ErrorAction SilentlyContinue
Write-Host "Word/Excel token cache cleared. Reopen and sign in."

OneDrive — Reset (see Step 3 below for full steps)

Stop-Process -Name OneDrive -Force -ErrorAction SilentlyContinue
& "$env:LOCALAPPDATA\Microsoft\OneDrive\OneDrive.exe" /reset
Start-Sleep -Seconds 10
Start-Process "$env:LOCALAPPDATA\Microsoft\OneDrive\OneDrive.exe"
Write-Host "OneDrive reset. Sign in with new credentials."

4. Step 3 — Fix OneDrive Sync Crash & Performance

OneDrive is the most common performance offender after an Okta password change. It loops trying to refresh a stale token and can consume 30–80% CPU.

Step-by-Step Fix

StepAction
1Right-click the OneDrive cloud icon in the system tray
2Click Settings > Account > Unlink this PC
3Confirm — this stops the sync loop immediately
4Open Task Manager and end any remaining OneDrive.exe processes
5Clear the OneDrive token cache (see PowerShell below)
6Reopen OneDrive from Start menu
7Sign in with the new credentials when prompted
8Allow OneDrive to re-sync (files are not deleted — they remain in the cloud)

Stop OneDrive + Clear Token Cache (PowerShell — Run as Admin)

# Stop OneDrive
Stop-Process -Name OneDrive -Force -ErrorAction SilentlyContinue

# Clear OneDrive token/auth cache
Remove-Item -Path "$env:LOCALAPPDATA\Microsoft\OneDrive\settings" -Recurse -Force -ErrorAction SilentlyContinue
Remove-Item -Path "$env:LOCALAPPDATA\Microsoft\OneDrive\logs" -Recurse -Force -ErrorAction SilentlyContinue

# Restart OneDrive
Start-Process "$env:LOCALAPPDATA\Microsoft\OneDrive\OneDrive.exe"
Write-Host "OneDrive restarted. Sign in with new credentials."

If OneDrive is Still Crashing or Looping

# Full OneDrive reset (keeps files, resets app state)
& "$env:LOCALAPPDATA\Microsoft\OneDrive\OneDrive.exe" /reset
Start-Sleep -Seconds 10
Start-Process "$env:LOCALAPPDATA\Microsoft\OneDrive\OneDrive.exe"
Write-Host "OneDrive reset complete. Sign in with new credentials."

Note: /reset does not delete cloud files. It resets the local sync engine state and clears all cached tokens.

5. Step 4 — Clear Stale Credentials from Windows Credential Manager

Why this matters: After an Okta password change, Microsoft apps (Office, Teams, OneDrive) cache an OAuth bearer token in Windows Credential Manager. That token is tied to the old password — once Okta invalidates it, apps silently fail to authenticate. Clearing the cached entries forces a fresh OAuth handshake so apps re-authenticate against Okta with the new credentials.

Windows Credential Manager stores cached passwords and tokens. After an Okta password change, old entries cause repeated auth failures.

Manual Steps

StepAction
1Open Control Panel > Credential Manager
2Click Windows Credentials
3Look for entries containing: MicrosoftOffice, OneDrive, Okta, MicrosoftAccount, live.com
4Click each one and select Remove
5Reboot or restart the affected apps

PowerShell — Remove Microsoft/Okta Credential Entries

# List all Microsoft/Okta credentials stored in Credential Manager
cmdkey /list | Where-Object { $_ -match "Microsoft|Okta|OneDrive|live" }

# Remove all MicrosoftOffice16 cached credentials
$creds = cmdkey /list | Select-String "MicrosoftOffice16"
foreach ($c in $creds) {
    $target = ($c -split "Target: ")[1].Trim()
    cmdkey /delete:$target
}
Write-Host "Stale Office credentials removed."

CheckExpected Result
Office app opens without sign-in promptToken accepted — auth is working
OneDrive cloud icon shows green checkmarkSync is active and healthy
CPU usage back to normal (< 10% idle)OneDrive sync loop resolved
Teams shows online statusTeams token refreshed

If CPU is still high after completing all steps, reboot the machine. A reboot clears all in-memory token caches and forces a clean re-auth.

8. Hybrid Environment — Okta + Microsoft Entra ID (Azure AD)

Applies to: Organizations using Okta as the primary IdP with Microsoft Entra ID (formerly Azure AD) in a hybrid join or federated configuration. Password changes propagate from Okta → Entra ID → on-prem AD (if synced via Entra Connect).

Why Hybrid Makes Token Issues Worse

LayerWhat Happens After Password Change
OktaNew session token issued immediately
Entra ID (cloud)Syncs from Okta via OIDC federation — can take 5–15 minutes
On-prem ADSyncs via Entra Connect — can take 15–30 minutes
Office appsHold cached Entra ID tokens — stale until cleared
OneDriveAuthenticates against Entra ID — loops when token mismatches
Windows credential cacheHolds both Okta and Entra ID tokens separately

Step 1 — Check Entra ID Sync Status (PowerShell)

# Connect to Microsoft Graph (run once)
Connect-MgGraph -Scopes "User.Read.All"

# Check user's last sync time and account enabled status
Get-MgUser -UserId "user@domain.com" | Select-Object DisplayName, UserPrincipalName, AccountEnabled, OnPremisesSyncEnabled, OnPremisesLastSyncDateTime

If OnPremisesSyncEnabled is True, the password change must propagate through Entra Connect before desktop apps will accept the new token. Wait for sync or force it (see below).

Step 2 — Force Entra Connect Sync (Run on Entra Connect Server)

# Force a delta sync from on-prem AD to Entra ID
Import-Module ADSync
Start-ADSyncSyncCycle -PolicyType Delta

# Or full sync if delta doesn't resolve it
Start-ADSyncSyncCycle -PolicyType Initial

Run this on the server running Entra Connect (Azure AD Connect). After sync completes (1–2 minutes), the new password hash propagates to Entra ID, and Office apps + OneDrive will accept new credentials.

Step 3 — Re-Register Windows Device with Entra ID (if Hybrid Joined)

# Check current Entra ID / Azure AD join status
dsregcmd /status

Look for: AzureAdJoined : YES (cloud joined), DomainJoined : YES (hybrid), AzureAdPrt : YES (Primary Refresh Token active).

If AzureAdPrt is NO after a password change:

# Force PRT refresh — run as the affected user (NOT admin)
dsregcmd /refreshprt

If that fails, sign out and back in to Windows with the new password to refresh the PRT.

Step 4 — Re-Authenticate OneDrive in Hybrid Environment

In a hybrid setup, OneDrive authenticates against Entra ID, not Okta directly. After password change:

StepAction
1Wait for Entra Connect sync to complete (or force it — see Step 2)
2Unlink OneDrive: system tray > Settings > Account > Unlink this PC
3Run the OneDrive reset PowerShell (see Step 3 in main SOP)
4Sign back in to OneDrive with the Microsoft/Entra ID account (not Okta URL directly)
5Verify green checkmark in tray within 2–3 minutes

Step 5 — Verify Hybrid Token Chain is Clean

# Confirm PRT (Primary Refresh Token) is valid
dsregcmd /status | findstr /i "AzureAdPrt"

# Confirm Entra ID user is enabled and synced
Get-MgUser -UserId "user@domain.com" | Select-Object AccountEnabled, OnPremisesLastSyncDateTime, PasswordPolicies
Expected OutputMeans
AzureAdPrt : YESDevice has valid Entra ID session token
AccountEnabled : TrueAccount not locked in Entra ID
OnPremisesLastSyncDateTime within last 30 minSync is current

Summary: After an Okta password change — always start at the Okta dashboard, launch Office apps from the app tile to pass a fresh token, unlink and re-link OneDrive, and clear stale credentials from Windows Credential Manager. A reboot after completing all steps ensures a clean slate.

9. Okta Admin — Bulk CSV User Import

Use CSV import to provision multiple users at once — new hires, acquisitions, or directory migrations. Okta's built-in CSV import maps columns to user profile attributes and supports group assignment and password staging.

When to Use CSV Import vs. Other Methods

MethodBest For
CSV ImportOne-time bulk adds, acquisitions, pilot groups — no AD sync available
Okta AD AgentOngoing sync from on-prem Active Directory
SCIM ProvisioningAutomated lifecycle management from HR system (Workday, BambooHR)
Okta APIScripted user creation with complex attribute mapping
Manual (Admin Console)Single users only — not scalable for bulk

Step 1 — Prepare the CSV File

Okta's CSV import expects specific column headers. Use the template below. firstName, lastName, and email are the minimum required fields.

firstName,lastName,email,login,mobilePhone,department,title,manager,groups
Jane,Smith,jsmith@company.com,jsmith@company.com,555-100-1001,IT,Systems Admin,manager@company.com,IT-Staff
John,Doe,jdoe@company.com,jdoe@company.com,555-100-1002,Finance,Analyst,manager@company.com,Finance-Team
ColumnNotes
firstName / lastNameRequired. Used for display name in Okta.
emailRequired. Primary email and used as the Okta username if login is omitted.
loginOkta username (usually same as email). Must be unique. If blank, Okta uses email.
mobilePhoneUsed for SMS MFA enrollment. Format: 555-100-1001 or +15551001001.
departmentMaps to Okta profile department attribute. Used for group rules.
titleJob title. Optional.
managerManager's email. Optional — used for org chart and approval routing.
groupsComma-separated Okta group names. Groups must already exist in Okta.

Encoding note: Save the CSV as UTF-8 (no BOM). In Excel: File → Save As → CSV UTF-8 (Comma delimited). BOM-encoded files can cause import errors.

Step 2 — Pre-Import Validation (PowerShell)

Run this before uploading to catch duplicates, missing fields, and malformed emails.

# Validate a CSV before Okta import
$csv = Import-Csv "C:\Temp\okta-import.csv"

# Check for missing required fields
$csv | Where-Object { -not $_.firstName -or -not $_.lastName -or -not $_.email } |
  Select-Object firstName, lastName, email |
  Format-Table

# Check for duplicate emails
$csv | Group-Object email | Where-Object { $_.Count -gt 1 } |
  Select-Object Name, Count

# Check for invalid email format
$emailPattern = '^[\w\.\-]+@[\w\.\-]+\.\w{2,}$'
$csv | Where-Object { $_.email -notmatch $emailPattern } |
  Select-Object firstName, lastName, email

Write-Host "Validation complete. Fix issues above before uploading."

Step 3 — Import via Okta Admin Console

StepAction
1Log in to Okta Admin Console
2Go to Directory → People
3Click More Actions → Import Users from CSV
4Click Browse and select your prepared CSV file
5Okta displays a column mapping preview — verify each column maps to the correct Okta attribute
6If columns don't auto-map, use the dropdowns to assign them manually
7Choose a Password Option: Set by admin, User must change at next login, or Send activation email
8Click Import

Step 4 — Review Import Results & Confirm

StepAction
1Okta shows a results summary: Users to Import, Warnings, Errors
2Review the Warnings column — these are users that will be skipped or partially imported
3Review the Errors column — these are rows Okta cannot process (duplicate login, invalid email, missing required field)
4For each error row, fix the CSV and re-import just those rows
5If results look correct, click Confirm Import
6Okta creates users in Staged status (not yet active)

Step 5 — Activate Users & Send Welcome Email

StepAction
1Directory → People → filter by Status: Staged
2Select all staged users (checkbox in header row)
3Click More Actions → Activate
4Choose Send activation email now to email users their activation link, or Activate without email to set passwords manually
5Confirm — users move to Active status and receive activation email (if selected)

The activation email contains a time-limited link (72 hours by default). If a user's link expires, go to their profile → More Actions → Re-send Activation Email.

Step 6 — Verify Imported Users

# Check import via Okta API — list users created in the last 24 hours
# Requires Okta API token (Settings → API → Tokens)
$headers = @{ Authorization = "SSWS YOUR_API_TOKEN" }
$since = (Get-Date).AddHours(-24).ToString("o")
$url = "https://yourcompany.okta.com/api/v1/users?filter=created+gt+`"$since`""
Invoke-RestMethod -Uri $url -Headers $headers | Select-Object @{n='Name';e={$_.profile.displayName}}, @{n='Email';e={$_.profile.email}}, status | Format-Table

Common Import Errors & Fixes

ErrorCause & Fix
Duplicate login / email already existsA user with that email already exists in Okta. Remove from CSV or deactivate the existing account first.
Invalid email formatEmail has spaces, special characters, or missing domain. Fix the email field in the CSV row.
Group not foundThe group name in the groups column doesn't match an existing Okta group (case-sensitive). Create the group first or correct the spelling.
Column mapping error — attribute not foundColumn header doesn't match any Okta profile attribute. Use the manual mapping dropdowns in Step 3, or rename the CSV column to match the Okta attribute name exactly.
CSV encoding error — garbled namesFile was saved as UTF-16 or ANSI. Re-save as UTF-8 (no BOM) from Excel or Notepad++.
Users stuck in Staged statusManually activate them: Directory → People → filter Staged → select all → Activate.
Activation email not receivedCheck spam. If expired: user profile → More Actions → Re-send Activation Email. Verify the email address in the profile is correct.
MFA enrollment not triggered after activationCheck the user's assigned applications and the MFA policy for those apps. The MFA prompt fires on first app access, not at activation.
