GraphQL vs SQL — Parsing & Response Formats

Dev Lansweeper GraphQL

1. Why GraphQL Responses Are Nested

GraphQL isn't a bug — it's a feature you didn't ask for yet. The response shape mirrors the query shape exactly. That's by design. But it means every level of your query adds a level of nesting to the response.

The Query Shape Creates the Response Shape

# This query...
{
  Site {
    site {
      assetBasicInfo(where: ...) {
        name
        ipAddress
        userName
      }
    }
  }
}

# ...produces this response:
{
  "data": {
    "Site": {
      "site": {
        "assetBasicInfo": [
          { "name": "PC-001", "ipAddress": "10.0.0.5", "userName": "jsmith" }
        ]
      }
    }
  }
}

You have to navigate data → Site → site → assetBasicInfo — four levels — before you reach the array you actually want. SQL hands you the array immediately.

Why Lansweeper's GraphQL Is Especially Deep

LevelWhat it represents
dataGraphQL standard wrapper — every response has this
data.SiteLansweeper's top-level scope — their schema design choice
data.Site.siteMulti-site support — the site object (even if you only have one)
data.Site.site.assetBasicInfoThe actual data array you wanted

Three of those four levels are just Lansweeper's schema structure. The data is fine — you just need to know the path to it.

2. SQL vs GraphQL — Response Shape Comparison

SQL — Flat Rows (Clean to Parse)

# Query
SELECT a.Name, a.IPAddress, ac.Username
FROM tblAssets a
JOIN tblAssetCustom ac ON a.AssetID = ac.AssetID

# Response — already flat, ready to use
[
  { "Name": "PC-001", "IPAddress": "10.0.0.5", "Username": "jsmith" },
  { "Name": "PC-002", "IPAddress": "10.0.0.6", "Username": "ajones" }
]

# Parse it — one line
for row in results:
    print(row["Name"], row["IPAddress"])

GraphQL — Nested (Needs Navigation)

# Same data, same fields — GraphQL response
{
  "data": {
    "Site": {
      "site": {
        "assetBasicInfo": [
          { "name": "PC-001", "ipAddress": "10.0.0.5", "userName": "jsmith" },
          { "name": "PC-002", "ipAddress": "10.0.0.6", "userName": "ajones" }
        ]
      }
    }
  }
}

# Parse it — navigate 4 levels first
assets = data["data"]["Site"]["site"]["assetBasicInfo"]
for a in assets:
    print(a["name"], a["ipAddress"])

Side-by-Side Summary

FactorSQLGraphQL
Response structureFlat rows — immediate arrayNested object — navigate to array
Parse complexityOne level — row["Field"]Four+ levels — data["data"]["Site"]["site"]["assetBasicInfo"]
Null fieldsExplicit NULL in SQLField may be missing entirely or null
CSV/Excel exportTrivial — rows map directlyMust flatten first
Joins / relationsFlat after JOIN — still easyNested objects within objects — harder
Field namesYour column aliases — you control themSchema names — Lansweeper's naming (camelCase)
Requires DB accessYes — network access to SQL ServerNo — HTTP API, works from anywhere
Works remotely / VPNOnly if SQL port is reachableYes — HTTPS, works anywhere with a PAT

The trade-off: SQL is cleaner to parse but requires direct database access. GraphQL works remotely over HTTPS but the response needs navigation. Both return the same data.

3. Flatten GraphQL Responses — Python

The fix is simple: extract the deep path once at the top, assign it to a variable, then work with the flat array like SQL rows.

The One-Line Fix

import requests

r = requests.post(LS_URL, json={"query": QUERY}, headers={"Authorization": f"Bearer {PAT}"})
data = r.json()

# Navigate once — assign to a clean variable
assets = data["data"]["Site"]["site"]["assetBasicInfo"]

# Now it's flat — parse exactly like SQL rows
for a in assets:
    print(a["name"], a.get("ipAddress","—"), a.get("userName","—"))

Safe Navigation — Handle Missing Fields

GraphQL fields can be null or absent if the asset hasn't reported that data. Use .get() with a default:

# Risky — crashes if userName is null
print(a["userName"].upper())

# Safe — .get() returns default if field is missing or null
name     = a.get("name", "Unknown")
ip       = a.get("ipAddress", "No IP")
user     = a.get("userName") or "Unassigned"   # handles None AND empty string
last_seen = a.get("lastSeen", "Never")
os_name  = a.get("operatingSystem", "Unknown OS")

Flatten to a List of Dicts (SQL-style)

# Convert GraphQL response to flat row format — same shape as SQL output
def flatten_ls_assets(gql_response):
    try:
        raw = gql_response["data"]["Site"]["site"]["assetBasicInfo"]
    except (KeyError, TypeError):
        return []
    return [
        {
            "name":     a.get("name", ""),
            "ip":       a.get("ipAddress", ""),
            "user":     a.get("userName") or "",
            "os":       a.get("operatingSystem", ""),
            "last_seen": a.get("lastSeen", ""),
        }
        for a in (raw or [])
    ]

# Use it
assets = flatten_ls_assets(r.json())
for a in assets:
    print(a["name"], a["ip"], a["user"])

Export Flattened GraphQL to CSV

import pandas as pd

assets = flatten_ls_assets(r.json())
pd.DataFrame(assets).to_csv("assets.csv", index=False)
# or
pd.DataFrame(assets).to_excel("assets.xlsx", index=False)

Handling Nested Objects Within GraphQL Results

Some Lansweeper queries return nested objects (e.g., software list within an asset). Flatten those with a loop:

# Query returns: asset → software[] within each asset
# Response shape:
# { "data": { "Site": { "site": {
#     "software": [
#       { "softwareName": "Teams", "softwareVersion": "1.6", "publisher": "Microsoft" }
#     ]
# } } } }

software_list = data["data"]["Site"]["site"].get("software", [])
rows = []
for s in software_list:
    rows.append({
        "app":       s.get("softwareName", ""),
        "version":   s.get("softwareVersion", ""),
        "publisher": s.get("publisher", ""),
    })
pd.DataFrame(rows).to_csv("software.csv", index=False)

4. Flatten GraphQL Responses — JavaScript (Clippy)

Same principle in JS — navigate once, then use the flat array. This is the pattern inside clippy.js for Lansweeper calls.

Current Clippy Pattern

// After fetch() resolves
const data = await r.json();

// Navigate the nesting — assign to flat variable
const assets = data?.data?.Site?.site?.assetBasicInfo ?? [];

// Now iterate cleanly
assets.forEach(a => {
    const name     = a.name     ?? "Unknown";
    const ip       = a.ipAddress ?? "No IP";
    const user     = a.userName  || "Unassigned";
    const lastSeen = a.lastSeen  ?? "Never";
    console.log(name, ip, user);
});

Optional Chaining (?.) — Why It Matters for GraphQL

The ?. operator short-circuits to undefined if any level is null/missing, instead of throwing a TypeError. Essential for GraphQL where any level can be absent:

// Without ?. — crashes if Site is null
const assets = data.data.Site.site.assetBasicInfo;  // TypeError!

// With ?. — safely returns undefined, ?? [] gives empty array fallback
const assets = data?.data?.Site?.site?.assetBasicInfo ?? [];

// Also safe on individual fields
const os = asset?.operatingSystem ?? "Unknown OS";

Build a Reusable Flatten Helper

// Put this once near the top of your script
function lsAssets(gqlData) {
    return (gqlData?.data?.Site?.site?.assetBasicInfo ?? []).map(a => ({
        name:     a.name     ?? "",
        ip:       a.ipAddress ?? "",
        user:     a.userName  || "",
        os:       a.operatingSystem ?? "",
        lastSeen: a.lastSeen  ?? "",
    }));
}

// Use it anywhere
const assets = lsAssets(await r.json());
assets.forEach(a => renderAssetCard(a));

Render Flat Assets as an HTML Table

function renderAssetsTable(assets) {
    if (!assets.length) return "
No assets found.
";
    const rows = assets.map(a =>
        `${a.name}${a.ip}${a.user || "—"}${a.lastSeen?.split("T")[0] ?? "—"}`
    ).join("");
    return `${rows}

        

            NameIPUserLast Seen
        

        
    
`;
}

5. When to Use Each — Decision Guide

SituationUseWhy
Working remotely / VPN / from Clippy in a browserGraphQLHTTPS only, no DB port needed
On-prem, direct DB access, writing reports for SSMSSQLFlat output, easier to paste into Excel
Exporting to CSV / Excel for a non-tech stakeholderSQLFlat rows — no flattening step
Real-time asset lookup from a web tool or scriptGraphQLAPI, no firewall issues, returns quickly
Complex joins across many Lansweeper tablesSQLGraphQL joins are limited; SQL JOINs are more flexible
Power Automate flow needs asset dataGraphQLPA HTTP connector works, no SQL Server connector needed
Need patch status, services, logon historySQLDeeper Lansweeper tables not all exposed in GraphQL
Quick lookup — one hostname, right nowGraphQLClippy's find HOSTNAME command uses this path

6. Lansweeper-Specific Patterns

The Standard Path — Memorize This

# Python — every Lansweeper GraphQL response starts here
assets = response["data"]["Site"]["site"]["COLLECTION_NAME"]

# JavaScript
const assets = data?.data?.Site?.site?.COLLECTION_NAME ?? [];

# COLLECTION_NAME is whatever you queried:
# assetBasicInfo, software, authorizedReports, ipAddresses, etc.

Common Collection Names & Their Paths

Data you wantGraphQL collection nameSQL table equivalent
Asset summary (name, IP, user, OS)assetBasicInfotblAssets + tblAssetCustom
Software installedsoftwaretblSoftware
IP addresses per assetipAddressestblNetworkAdapters
Authorized / saved reportsauthorizedReportsLansweeper report engine
Sites in your accountsites (top level)N/A — cloud concept

Multi-Site Setup — Slightly Different Path

# Single site (most setups) — standard path
assets = data["data"]["Site"]["site"]["assetBasicInfo"]

# Multi-site — iterate sites first
for site in data["data"]["Site"]["sites"]:
    site_name = site["name"]
    assets    = site.get("assetBasicInfo", [])
    for a in assets:
        print(site_name, a["name"], a["ipAddress"])

Error Checking — GraphQL Returns 200 Even on Errors

Unlike REST APIs, GraphQL always returns HTTP 200. Errors are inside the response body under an errors key:

data = r.json()

# Always check for errors before navigating
if "errors" in data:
    for err in data["errors"]:
        print("GraphQL error:", err["message"])
else:
    assets = data["data"]["Site"]["site"]["assetBasicInfo"]
    # ... process assets

Debugging — Pretty Print the Full Response

import json

# When parsing feels wrong — print the whole thing formatted
r = requests.post(LS_URL, json={"query": QUERY}, headers={"Authorization": f"Bearer {PAT}"})
print(json.dumps(r.json(), indent=2))
