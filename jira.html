IT Help Desk Overview

NewsBank uses Jira Cloud (Software Development / Service Management) for IT Help Desk ticketing. All help desk requests are tracked in the HELP project. Issues are formatted as HELP-XXXX (e.g. HELP-1042).

Key Details

ItemValue
Jira typeJira Cloud (Atlassian hosted)
Project keyHELP
Issue formatHELP-XXXX
Helpdesk emailhelpdesk@newsbank.com
Purchasing emailpurchasing@newsbank.com
Default issue typeTask
API base URLhttps://[YOUR_DOMAIN].atlassian.net/rest/api/3
Auth methodBasic Auth (email + API token)

Who Uses What

  • End users (customers): Submit requests via the Jira customer portal or by emailing helpdesk@newsbank.com
  • IT agents: View and work tickets from the agent/board view
  • Power Automate flows: Create, update, transition, and comment on tickets via the REST API
  • Clippy work mode: Queries open tickets using JQL via the REST API
Note: Customer portal submissions and agent-created issues both land in the HELP project but may show different fields. See Cloud Sync Issues if custom fields aren't visible on both sides.

Create a New Ticket

There are three ways to create a HELP project issue.

Method 1 — Customer Portal (end user)

  1. Go to your-domain.atlassian.net/servicedesk/customer/portal/<ID>
  2. Choose the request type (e.g. "Get IT Help")
  3. Fill in the summary and description → Submit
  4. Jira creates a HELP-XXXX issue and emails a confirmation to the requestor

Method 2 — Email to helpdesk@newsbank.com

Jira can be configured to auto-create an issue from inbound email. When set up, any email to helpdesk@newsbank.com creates a HELP issue with the email subject as the summary and body as the description.

Method 3 — REST API (automated flows)

Used by the PO_Jira_Create_Issue_Flow Power Automate flow on PO submission. See PO Tracking Setup.

POST https://[YOUR_DOMAIN].atlassian.net/rest/api/3/issue
Authorization: Basic <base64(email:apitoken)>
Content-Type: application/json

{
  "fields": {
    "project": { "key": "HELP" },
    "summary": "PO #2025-042 - Request to Purchase Equipment - Jane Smith",
    "issuetype": { "name": "Task" },
    "priority": { "name": "Medium" },
    "description": {
      "type": "doc",
      "version": 1,
      "content": [
        { "type": "paragraph", "content": [{ "type": "text", "text": "PO details here." }] }
      ]
    },
    "labels": ["purchase-order", "pending-approval"]
  }
}
Response (201 Created): Jira returns { "id": "10042", "key": "HELP-42", "self": "https://..." }. Save the key value — this is your HELP-XXXX reference.

Assign and Prioritize a Ticket

Once a HELP issue is created, an IT agent assigns it and sets priority.

In the Jira UI

  1. Open the issue (e.g. HELP-42)
  2. Click the Assignee field → select the IT agent responsible
  3. Click Priority → choose High / Medium / Low
  4. Optionally set a Due date for SLA tracking

Priority Guidelines

PriorityWhen to usePO flow auto-set
HighUrgent / blocking work, PO ≥ $1,000Total amount ≥ $1,000
MediumStandard requests, PO $500–$999Total amount $500–$999
LowNon-urgent, PO < $500Total amount < $500

Via API (update priority and assignee)

PUT https://[YOUR_DOMAIN].atlassian.net/rest/api/3/issue/HELP-42
Authorization: Basic <base64(email:apitoken)>
Content-Type: application/json

{
  "fields": {
    "assignee": { "accountId": "5b10a2844c20165700ede21g" },
    "priority": { "name": "High" }
  }
}
Getting accountId: Use GET /rest/api/3/user/search?query=firstname.lastname to look up a user's accountId.

Transition Ticket Status

Jira workflow transitions move an issue from one status to another. Each transition has a numeric ID that's required for API calls.

Typical HELP Project Workflow

To Do → In Progress → Done
                   ↘ Rejected / Closed

In the Jira UI

  1. Open the issue
  2. Click the status dropdown in the top-right area of the issue (e.g. "To Do")
  3. Select the new status — Jira moves it and records the transition

Via API

Transitions require the numeric transition ID. See Find Workflow Transition IDs to get yours.

POST https://[YOUR_DOMAIN].atlassian.net/rest/api/3/issue/HELP-42/transitions
Authorization: Basic <base64(email:apitoken)>
Content-Type: application/json

{
  "transition": {
    "id": "21"
  }
}
Success: HTTP 204 No Content — no body returned. The issue status has changed.
400 No transition found: The transition ID is wrong, or the issue is already in a state that doesn't allow that transition. Verify with the transitions API.

Get Your API Token Jira Cloud

All automated calls (Power Automate flows, Clippy work mode) authenticate using a Jira Cloud API token tied to a service account email. This is different from a password — it's a token generated at Atlassian's identity portal.

Steps

  1. Log in to Jira using the service account (e.g. svc-jira@newsbank.com or the IT admin account that will own the automation)
  2. Go to https://id.atlassian.com → Security tab
  3. Under "API tokens", click Create and manage API tokens
  4. Click Create API token
  5. Label: Power Automate PO Sync (or similar)
  6. Click Create → copy the token immediately — it is only shown once
  7. Note the service account email address — you need both the email and the token

Store in Azure Key Vault

Go to Azure Portal → Key Vaults → po-system-vault → Secrets → + Generate/Import and add:

Secret NameValue
jira-api-tokenThe token copied from Atlassian
jira-service-account-emailThe service account email (e.g. svc-jira@newsbank.com)
Token expiry: Atlassian API tokens do not expire by default, but can be revoked. If flows start returning 401, the token may have been revoked — generate a new one and update the Key Vault secret.

Build Basic Auth Header for API Calls

Jira Cloud REST API uses HTTP Basic Authentication. The header value is a base64-encoded string of email:apitoken.

Format

Authorization: Basic <base64(email:apitoken)>

Example — In Power Automate (Compose action)

Basic @{base64(concat(body('Get_Jira_Service_Account_Email')?['value'], ':', body('Get_Jira_API_Token')?['value']))}

This reads both Key Vault secrets and builds the header value in one expression.

Example — Manual test (PowerShell)

# Build the header manually for testing
$email = "svc-jira@newsbank.com"
$token = "your-api-token-here"
$bytes = [System.Text.Encoding]::UTF8.GetBytes("$email`:$token")
$base64 = [System.Convert]::ToBase64String($bytes)
$headers = @{ Authorization = "Basic $base64"; "Content-Type" = "application/json" }

# Test: get your own account info
Invoke-RestMethod -Uri "https://[YOUR_DOMAIN].atlassian.net/rest/api/3/myself" -Headers $headers

Example — curl (bash/terminal)

curl -u "svc-jira@newsbank.com:your-api-token" \
  -H "Accept: application/json" \
  "https://[YOUR_DOMAIN].atlassian.net/rest/api/3/myself"
Note: Always use HTTPS. Never hard-code the token in flow definitions — pull it from Azure Key Vault at runtime.

REST API: Search Issues (JQL)

The core Jira API call for fetching tickets. Used by Clippy work mode to show your open tickets.

Endpoint

GET https://[YOUR_DOMAIN].atlassian.net/rest/api/3/search
  ?jql=<jql-query>
  &maxResults=10
  &fields=summary,status,priority,assignee,created,issuetype

Key Parameters

ParameterDescriptionExample
jqlJQL filter stringproject=HELP AND status!="Done"
maxResultsMax issues returned (default 50, max 100)25
fieldsComma-separated fields to returnsummary,status,priority
startAtOffset for pagination (0-based)0

Common JQL Queries

Use CaseJQL
My open ticketsassignee = currentUser() AND status != Done ORDER BY updated DESC
All HELP tickets openproject = HELP AND status != Done
Purchase order ticketsproject = HELP AND labels = purchase-order
Created in last 7 daysproject = HELP AND created >= -7d ORDER BY created DESC
High priority openproject = HELP AND priority = High AND status != Done

Example Response (abbreviated)

{
  "total": 12,
  "issues": [
    {
      "key": "HELP-42",
      "fields": {
        "summary": "PO #2025-042 - Request to Purchase Equipment - Jane Smith",
        "status": { "name": "In Progress" },
        "priority": { "name": "High" },
        "assignee": { "displayName": "IT Admin" }
      }
    }
  ]
}
Clippy work mode: When you type "my tickets" in Clippy (after work config is set), it runs assignee = currentUser() AND status != Done ORDER BY updated DESC and displays the results as ticket cards.

REST API: Update Issue Custom Fields

Used by the Lansweeper Device Sync flow to write device data (hostname, OS, serial number, etc.) into Jira custom fields after a ticket is created.

Endpoint

PUT https://[YOUR_DOMAIN].atlassian.net/rest/api/3/issue/{issueKey}
Authorization: Basic <base64(email:apitoken)>
Content-Type: application/json

Example — Update custom fields

{
  "fields": {
    "customfield_10100": "LAPTOP-001",
    "customfield_10101": "Windows 11 Pro",
    "customfield_10102": "SN-ABC123456",
    "customfield_10103": "Dell Latitude 5540",
    "summary": "Updated summary (optional)"
  }
}
Finding field IDs: Replace customfield_10100 with your actual field IDs. See Find Custom Field IDs to discover your field numbers.

Example — Update standard fields

{
  "fields": {
    "priority": { "name": "High" },
    "labels": ["purchase-order", "approved"],
    "assignee": { "accountId": "5b10a2844c20165700ede21g" }
  }
}
Success: HTTP 204 No Content. No body is returned on a successful update.

REST API: Post a Comment

Jira Cloud requires comments to be in Atlassian Document Format (ADF) — a JSON structure, not plain HTML or markdown. The PO approval flow posts approval/rejection comments using ADF.

Endpoint

POST https://[YOUR_DOMAIN].atlassian.net/rest/api/3/issue/{issueKey}/comment
Authorization: Basic <base64(email:apitoken)>
Content-Type: application/json

Minimal ADF — Plain paragraph

{
  "body": {
    "type": "doc",
    "version": 1,
    "content": [
      {
        "type": "paragraph",
        "content": [
          { "type": "text", "text": "This PO has been approved." }
        ]
      }
    ]
  }
}

ADF with Bold + Color (approval comment style)

{
  "body": {
    "type": "doc",
    "version": 1,
    "content": [
      {
        "type": "paragraph",
        "content": [
          {
            "type": "text",
            "text": "APPROVED",
            "marks": [
              { "type": "strong" },
              { "type": "textColor", "attrs": { "color": "#006644" } }
            ]
          },
          { "type": "text", "text": " — PO #2025-042 approved by John Manager on 2025-05-13." }
        ]
      },
      {
        "type": "paragraph",
        "content": [
          { "type": "text", "text": "Approval notes: ", "marks": [{ "type": "strong" }] },
          { "type": "text", "text": "Standard equipment request, approved." }
        ]
      }
    ]
  }
}

Find Custom Field IDs

Before you can read or write Jira custom fields via the API, you need to know their numeric IDs (e.g. customfield_10100). Custom fields are created by Jira admins and vary per instance.

API call to list all fields

GET https://[YOUR_DOMAIN].atlassian.net/rest/api/3/field
Authorization: Basic <base64(email:apitoken)>
Accept: application/json

Example response (excerpt)

[
  {
    "id": "summary",
    "name": "Summary",
    "schema": { "type": "string", "system": "summary" }
  },
  {
    "id": "customfield_10100",
    "name": "Hostname",
    "custom": true,
    "schema": { "type": "string", "custom": "com.atlassian.jira.plugin.system.customfieldtypes:textfield" }
  },
  {
    "id": "customfield_10101",
    "name": "Operating System",
    "custom": true,
    "schema": { "type": "string", "custom": "..." }
  }
]

In Power Automate (HTTP action)

  1. Add an HTTP action → Method: GET
  2. URI: https://[YOUR_DOMAIN].atlassian.net/rest/api/3/field
  3. Headers: Authorization: Basic @{...}, Accept: application/json
  4. Run the flow once and look at the output in Run History
  5. Ctrl+F for the field name you created (e.g. "Hostname") — the id next to it is your field ID
Custom field IDs are instance-specific. The IDs in the Lansweeper Device Sync flow use placeholder names like customfield_HOSTNAME. You must replace these with your actual IDs once you run the field list call from your Jira instance.

Find Workflow Transition IDs

To transition a Jira issue via the API (e.g. move from "Pending" to "Approved"), you need the numeric transition ID for your project's workflow. These IDs are unique per Jira instance and project.

API call to list transitions for an issue

GET https://[YOUR_DOMAIN].atlassian.net/rest/api/3/issue/HELP-1/transitions
Authorization: Basic <base64(email:apitoken)>
Accept: application/json

Replace HELP-1 with any existing issue in your HELP project.

Example response

{
  "transitions": [
    { "id": "11", "name": "To Do" },
    { "id": "21", "name": "In Progress" },
    { "id": "31", "name": "Done" },
    { "id": "41", "name": "Rejected" }
  ]
}

Record your transition IDs

Workflow StateTransition NameTransition ID
PO Approved — ready to order(e.g. "In Progress")___
PO Rejected — closed(e.g. "Done" or "Rejected")___

Paste IDs into the flow

In PO_Manager_Approval_Workflow.json, find and replace:

  • [APPROVED_TRANSITION_ID] → your approved transition ID (e.g. 21)
  • [REJECTED_TRANSITION_ID] → your rejected transition ID (e.g. 41)
Workflow restrictions: Some Jira workflows only allow certain transitions from specific states. If you get a 400 error saying "No transition found", confirm the issue is in the expected starting state before the transition is attempted.

PO Approval Tracking Setup

Purchase Orders submitted through the PowerApp create a HELP-XXXX issue in Jira and are tracked from submission through approval or rejection.

End-to-End Flow

User submits PO in PowerApp
  └── PO_Jira_Create_Issue_Flow
        ├── Creates HELP-XXXX (status: To Do / Pending)
        ├── Sets priority based on total amount
        ├── Labels: ["purchase-order", "pending-approval"]
        └── Returns jiraIssueKey + jiraIssueUrl → saved to SharePoint PO list
              └── PO_Manager_Approval_Workflow
                    ├── Sends approval request to manager
                    ├── CC: requestor + purchasing@newsbank.com + any additional CC
                    ├── APPROVED
                    │     ├── Transitions HELP-XXXX → approved state
                    │     ├── Posts green "APPROVED" comment
                    │     └── Emails requestor (CC: manager + purchasing@newsbank.com)
                    └── REJECTED
                          ├── Transitions HELP-XXXX → rejected/done state
                          ├── Posts red "REJECTED" comment
                          └── Emails requestor (CC: manager)

What Purchasing Sees in Jira

When a PO is approved, the HELP project issue shows:

  • Summary: PO #2025-042 - Request to Purchase Equipment - Jane Smith
  • Status: Approved / In Progress
  • Labels: purchase-order, approved
  • Comment: APPROVED — details, notes, total amount, shipping
  • Description: Full PO table with all line items

Purchasing should action the order and update the Jira issue when ordered and again when received.

SharePoint PO List

The JiraIssueKey column (single line of text) on the Purchase Orders SharePoint list stores the HELP-XXXX key for cross-reference. IT admins can look up a PO by SharePoint ID and jump directly to the Jira issue.

Lansweeper Device Sync Setup

When a new Jira HELP issue is created, the Jira_Lansweeper_Device_Sync_Flow automatically looks up the requestor's device in Lansweeper and populates Jira custom fields with device data — no manual data entry needed.

How It Works

  1. Jira fires an issue_created webhook → HTTP POST to the Power Automate flow trigger URL
  2. Flow responds 200 immediately (Jira needs a fast response or it retries)
  3. Flow extracts the reporter's email (UPN) from the webhook payload
  4. Queries Lansweeper GraphQL: assetUsers.username = reporter@newsbank.com
  5. If a device is found: PUT to Jira to populate custom fields (hostname, OS, serial, model, warranty)
  6. Posts a comment on the issue: "Device data populated from Lansweeper"
  7. If no device found: posts a "no device found" comment so the agent knows to enter it manually

Custom Fields Populated

Jira Custom FieldLansweeper Source
HostnameassetBasicInfo.name
Operating SystemoperatingSystem.caption
OS BuildoperatingSystem.buildNumber
Serial NumberassetCustom.serialNumber
ModelassetCustom.model
Warranty ExpiryassetCustom.warrantyDate
DepartmentassetCustom.department
LocationassetCustom.location
IP AddressassetBasicInfo.ipAddress
Last SeenassetBasicInfo.lastSeen

Setup Steps

  1. Get Lansweeper API token and site ID (if not already in Key Vault)
  2. Get Jira API token (see API Token setup)
  3. Get your Jira custom field IDs (see Find Custom Field IDs)
  4. Import Jira_Lansweeper_Device_Sync_Flow.json into Power Automate
  5. Replace all customfield_HOSTNAME etc. placeholders with real field IDs
  6. Replace [YOUR_DOMAIN] with your Atlassian subdomain
  7. Set up the Jira webhook (see Webhook Setup)

See the full deployment SOP: SOPs/SOP_Jira_Lansweeper_Device_Sync.md

Webhook Setup for Power Automate

Jira webhooks fire an HTTP POST to a URL you specify when certain events occur. The Lansweeper sync flow uses a webhook on issue_created to trigger automatically when any HELP ticket is created.

Steps to Configure in Jira

  1. In Jira: Settings (gear icon) → System → WebHooks
  2. Click Create a WebHook
  3. Name: Lansweeper Device Sync
  4. URL: paste the HTTP POST trigger URL from your Power Automate flow (the URL shown after you add an HTTP request trigger)
  5. Events: check Issue → created
  6. JQL filter (optional but recommended): project = HELP — limits the webhook to HELP project only
  7. Click Create
Get the trigger URL: In Power Automate, open the Jira_Lansweeper_Device_Sync_Flow → edit mode → click the "When a HTTP request is received" trigger → copy the HTTP POST URL shown at the bottom of the trigger card. (The URL only appears after the flow has been saved at least once.)

Webhook Payload (what Jira sends)

{
  "webhookEvent": "jira:issue_created",
  "issue": {
    "key": "HELP-42",
    "fields": {
      "summary": "Laptop keyboard not working",
      "reporter": {
        "emailAddress": "jane.smith@newsbank.com",
        "displayName": "Jane Smith"
      },
      "status": { "name": "To Do" },
      "issuetype": { "name": "Task" }
    }
  }
}

Jira Webhook Delivery Notes

  • Jira expects a 200 response within ~10 seconds. If the flow takes longer (e.g. waiting on Lansweeper), it will retry the webhook
  • The sync flow is designed to respond 200 immediately and then do the work asynchronously to avoid this problem
  • Webhook delivery failures appear in Jira Settings → System → WebHooks → Failed deliveries

Cloud Sync Issues: Customer vs Agent Views

Jira Cloud separates the customer portal (what requestors see) from the agent view (what IT staff sees in the board/backlog). Data submitted in one view may not appear in the other — this is by design, but causes confusion during automation setup.

Common Issues and Fixes

1. Custom fields visible in agent view but not customer portal

Jira Service Management controls which fields appear in each portal request type. Custom fields added to the agent issue screen won't automatically appear in the customer portal form.

  • Fix: In Jira → Project settings → Request types → select the request type → add the custom field to the form
  • Alternatively: set custom fields via API after creation (this is what the Lansweeper sync does — customers don't enter device info, IT populates it automatically)

2. Fields submitted via customer portal missing in agent view

Customer portal fields use request type fields, not standard issue fields. Some fields submitted by customers may be stored as "additional properties" rather than standard Jira fields.

  • Fix: In Project settings → Fields → verify the field is mapped to both the agent screen and the portal request type

3. Webhook fires but reporter email is blank

If a ticket was created via the customer portal by an unregistered user or an anonymous submission, reporter.emailAddress may be null or a Jira-generated service account email.

  • Fix: Ensure the customer portal requires Jira account login (not anonymous), OR add a fallback in the sync flow to skip Lansweeper lookup if emailAddress is empty

4. Webhook fires twice for one ticket

This can happen if Jira triggers issue_created plus an immediate issue_updated for an auto-assigned field. The sync flow uses jira:issue_created filtering, so the second event is ignored — but check flow run history if you see duplicate comments on tickets.

5. Reporter UPN format mismatch (Jira vs Lansweeper)

The Lansweeper sync queries by assetUsers.username. In some environments, Lansweeper stores the domain prefix format (NEWSBANK\jsmith) rather than the UPN format (jsmith@newsbank.com).

  • Fix: Check the Lansweeper Users tab on a known asset — if names show as DOMAIN\username, update the GraphQL query in the sync flow to strip the domain prefix from the UPN for the comparison

6. Custom fields populated by API not visible in customer portal

The Lansweeper sync writes device data to Jira custom fields via the agent-facing API. These fields typically only appear in the agent view, which is the intended behavior — device data is for IT staff, not end users.

CORS Errors from Browser API Calls

Jira Cloud (and Lansweeper Cloud) actively block API calls made directly from browser JavaScript on domains they don't control. This affects Clippy when it tries to call the Jira or Lansweeper APIs directly.

What CORS Means Here

When Clippy (running at greenhornet-dev.github.io) makes a fetch request to [YOUR_DOMAIN].atlassian.net, the browser sends a preflight OPTIONS request. Atlassian's servers respond with CORS headers that block the request unless the origin is on their allowlist — and github.io is not.

Symptoms

Access to fetch at 'https://[domain].atlassian.net/rest/api/3/search' from origin
'https://greenhornet-dev.github.io' has been blocked by CORS policy:
No 'Access-Control-Allow-Origin' header is present on the requested resource.

The Fix: Route Through Power Automate

Power Automate makes the API call server-side — there's no browser CORS restriction when a flow calls an external HTTP endpoint. The pattern:

  1. Clippy calls a Power Automate HTTP trigger (which is on Microsoft's domain — no CORS issue)
  2. The PA flow receives the request, calls Jira or Lansweeper, and returns the result
  3. Clippy displays the result
Browser (Clippy)
    → POST https://prod-xx.westus.logic.azure.com/workflows/...   ← no CORS block
          ↓ Power Automate flow
          → GET https://[YOUR_DOMAIN].atlassian.net/rest/api/3/search  ← server-side, no CORS
          ← returns JSON result
    ← Clippy displays tickets

Clippy Error Message When CORS Blocks

When Clippy can't reach Jira directly, it displays:

CORS blocked: Jira Cloud doesn't allow direct browser API calls from this origin. Route through a Power Automate HTTP trigger instead — set your Jira URL in work config to point to a PA proxy flow.

Short-term Test Workaround (dev only)

During development you can disable CORS in Chrome using a flag or a CORS-proxy extension, but never use this in production. The proper fix is the Power Automate proxy pattern above.
